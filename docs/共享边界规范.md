# 五端互通共享边界规范

适用范围：Cursor / Antigravity / Claude Code / Codex / Gemini 五端互通场景。
目的：明确哪些数据必须共享、哪些必须隔离、哪些严禁外泄，统一通过 `ceo-console` 作为唯一数据中枢。

---

## 1. 角色与数据中枢

- 唯一事实源（SoT）：`ceo-console` 后端数据库 `data/ceo_console.db`
- 唯一访问入口：`http://127.0.0.1:5050/api/*`
- 五端不直接互相读写本地任务文件，任何任务/项目/状态变更都通过 API 提交

---

## 2. 字段级白名单（必须共享）

### 2.1 任务主数据
| 字段 | 类型 | 共享原因 |
| --- | --- | --- |
| `id` | int | 全局唯一标识 |
| `title` | string | 跨端任务识别 |
| `project` | string | 项目归属 |
| `priority` | enum (P0/P1/P2) | 调度优先级 |
| `status` | enum (待分配/AI执行中/待人工审查/已完成) | 统一状态机 |
| `due_at` | datetime | 截止时间 |
| `assignee_ai` | enum (Antigravity/Claude Code/Cursor/Codex/Gemini/Other) | 执行端归属 |
| `acceptance_criteria` | string | 验收标准 |
| `notes` | string | 任务备注 |

### 2.2 执行可追溯数据
| 字段 | 用途 |
| --- | --- |
| `execution_state` | idle/running/succeeded/failed/unsupported |
| `execution_tool` | 实际执行端（含自动回退记录） |
| `execution_command` | 实际命令（已脱敏） |
| `execution_progress` | 进度日志流 |
| `execution_error` | 失败原因 |
| `execution_started_at` / `execution_finished_at` | 时间线 |

### 2.3 闭环审查数据
| 字段 | 用途 |
| --- | --- |
| `review_result` | approved / rejected |
| `review_comment` | 审查意见或重开原因 |
| `reviewed_at` | 审查时间 |

### 2.4 项目元数据
- 项目列表与归档状态（`/api/projects`）
- 项目规则文件存在性：`has_ai_rule` / `has_cursorrules` / `has_coordinator` / `has_execution_checklist`

---

## 3. 建议共享（提升协作效率，非强制）

- 日报与预警：`/api/daily-brief`（超时、待审查、失败汇总）
- 看板汇总：`/api/dashboard-summary`
- 工具可用性快照：`/api/tools/status`
- 健康指标：`/api/health`
- 标准提示词模板（分派 / 审查 / 重开）

---

## 4. 端内私有（建议隔离，不写入中枢）

- 五端各自的会话上下文、短期记忆、临时草稿
- IDE 打开文件、光标位置、插件临时状态
- 个性化偏好：主题、快捷键、本地工作区配置
- 未提交的中间推理结果

> 这类数据不进入 `ceo-console`，避免污染主数据并降低跨端冲突风险。

---

## 5. 黑名单（严禁共享）

| 类型 | 示例 | 替代方案 |
| --- | --- | --- |
| API 密钥 / Token | OpenAI Key、OAuth Refresh Token | 统一走密管（1Password CLI `op`） |
| 账号凭据 | 密码、Cookie、会话票据 | 不写入数据库、不打印日志 |
| 高敏个人数据 | 证件号、银行卡号 | 必须脱敏或不入库 |
| 未脱敏生产数据 | 含 PII 的客户表 | 入库前脱敏，或仅存引用 ID |

强制要求：执行日志写入 `data/run-logs/` 前，密钥相关字段必须脱敏。

---

## 6. 统一状态机

```
[待分配] --(dispatch)--> [AI执行中] --(succeeded)--> [待人工审查]
                                                       │
                                                       ├──(approve)--> [已完成]
                                                       └──(reject)---> [待分配]
```

- 任意端只能通过 API 触发状态变更
- 禁止任何端在本地“私自完成”任务，必须经过 `待人工审查` 闭环

---

## 7. API 调用示例（跨端统一）

### 7.1 创建任务
```bash
curl -X POST http://127.0.0.1:5050/api/tasks \
  -H "Content-Type: application/json" \
  -d '{
    "title": "网关重构",
    "project": "edge-caculate-box",
    "assignee_ai": "Claude Code",
    "priority": "P1",
    "acceptance_criteria": "wavegate改名为src",
    "notes": "无"
  }'
```

### 7.2 调度执行
```bash
curl -X POST http://127.0.0.1:5050/api/tasks/<task_id>/dispatch
```

### 7.3 审查通过 / 驳回
```bash
curl -X POST http://127.0.0.1:5050/api/tasks/<task_id>/review \
  -H "Content-Type: application/json" \
  -d '{"decision": "approve", "comment": "验收通过"}'
```

---

## 8. 审计要求（必须遵守）

- 所有写入接口的调用须可追溯（含发起端、时间、参数）
- 删除项目 / 删除任务 / 重开 / 审查变更：必须有日志记录
- 执行日志保留路径：`ceo-console/data/run-logs/task-<id>-<ts>.log`
- 日志保留策略：默认保留 30 天，可按项目要求延长

---

## 9. 落地规则（强制）

1. **规则 1**：任务与审查走库，不走聊天记录
2. **规则 2**：执行日志可共享，但凭据必须脱敏
3. **规则 3**：跨端只认统一状态机，不认自由文本状态
4. **规则 4**：所有删除 / 重开 / 审查动作必须写审计日志
5. **规则 5**：密钥统一走密管，端侧不落盘
6. **规则 6**：新增字段需先评审并入此文档，再开放 API 写入

---

## 10. 例外审批

任何对本规范的临时例外（例如某项目需要短期共享私有上下文），必须：

- 由 CEO 决策并书面记录
- 在 `ceo-console/data/exceptions/` 留存例外条目
- 规定失效时间，到期自动回收
